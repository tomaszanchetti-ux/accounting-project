# ARD WS20 - 31/03/2026

## Objetivo de la sesion

Corregir el deploy del backend para que el MVP quede listo para una demo
publica end-to-end y cerrar el riesgo residual detectado tras el cierre de
`EPIC 09`.

## Diagnostico encontrado

- la URL publica del frontend en Vercel respondia `200`
- la URL objetivo del backend en Railway respondia `404` en `/health`
- el bloqueo real no era solo Railway:
  - el setup flow registraba `expected_totals`, `concept_master` y
    `employee_reference` como `local_path`
  - eso solo funciona cuando frontend y backend comparten filesystem
  - en Vercel + Railway esos paths no existen del lado del backend

## Cambios aplicados

### 1. Se corrigio el setup flow para deploy separado

Archivos:

- `frontend/lib/setup/demo-seed.ts`
- `frontend/app/page.tsx`
- `frontend/components/setup/setup-workspace.tsx`
- `frontend/components/setup/run-setup-form.tsx`

Resultado:

- los demo seed files ahora se leen en server-side desde el frontend
- al iniciar la run se suben al backend como archivos reales via multipart
- `reset expected totals` vuelve a subir el CSV demo en vez de registrar un path
- el flujo deja de depender de filesystem compartido entre servicios

### 2. Se dejo el backend listo para Railway desde el root del repo

Archivos:

- `Dockerfile`
- `.dockerignore`
- `railway.toml`
- `backend/app/core/settings.py`

Resultado:

- Railway puede construir el backend correcto desde el root del monorepo
- el servicio queda configurado con healthcheck en `/health`
- el proceso escucha `PORT` de Railway ademas de `API_PORT`

### 3. Se actualizo documentacion de deploy

Archivos:

- `README.md`
- `frontend/README.md`
- `docs/DEPLOY_BACKEND.md`

Resultado:

- la estrategia de deploy queda alineada con el fix real
- queda explicitado que el paso pendiente es operativo en Railway, no de codigo
